# World Labs Marble & RTFM：交互式 3D 世界模型

> **机构**：World Labs  **发布时间**：Marble 2025.11，RTFM 2025
> **形式**：官方技术博客（无正式论文）
> **来源**：https://worldlabs.ai/blog/marble-world-model  |  https://worldlabs.ai/blog/rtfm

---

## 一句话总结

World Labs 提出了一套双模块世界模型系统：**Marble** 负责从多模态输入生成持久化 3D 场景（基于 Gaussian Splats），**RTFM** 负责在此基础上以交互帧率实时渲染新视角（基于自回归扩散 Transformer）——两者分工协作，构成从静态感知到动态交互的完整闭环。

---

## 系统架构概览

World Labs 将世界模型定义为三类组件的闭环：

```
Renderers（渲染器） → Simulators（模拟器） → Planners（规划器）
       ↑_________________________________________________↓
```

在这一分类框架下，Marble 与 RTFM 各司其职、相互补充：

| 模块 | 定位 | 核心范式 | 主要职责 |
|------|------|----------|----------|
| **Marble** | 3D 世界生成器 | 3D Neural（基于 3DGS） | 多模态输入 → 持久化 3D 场景重建与编辑 |
| **RTFM** | 实时交互渲染器 | Generative（自回归扩散） | 3D 场景 → 交互式实时漫游与新视角合成 |

典型工作流：单张图像 → **Marble** 生成完整 3D 世界（3DGS 表示）→ **RTFM** 接管，实现实时交互漫游。两者结合，覆盖了从离线重建到在线交互的完整链路。

---

## Marble：3D 世界生成

### 场景表示（Enc 等价）

Marble 的核心场景表示格式为 **Gaussian Splats（3D 高斯泼溅，3DGS）**，将场景编码为半透明粒子的集合，兼具高保真渲染质量与灵活的几何操作能力。

为对接下游物理仿真与渲染引擎，Marble 支持将 3DGS 导出为两种 Triangle Mesh 格式：

- **Collider Meshes**：低保真网格，几何简化，专用于物理碰撞模拟（如机器人导航、刚体动力学）
- **High-Quality Meshes**：视觉质量接近原始 Gaussian Splats，适用于高保真渲染与资产导出

渲染引擎方面，World Labs 开源了 **Spark** 渲染器，基于 THREE.js 构建，支持跨平台部署（Web、桌面等）。

> **注**：3DGS 内部的编码网络架构（如编码器结构、训练目标、优化策略）均**未公开**。

### 多模态输入处理

Marble 支持五类输入模态，覆盖从纯文字到结构化 3D 布局的广泛场景：

| 输入类型 | 描述 | 典型用途 |
|---------|------|----------|
| **文字（Text）** | 自然语言场景描述 | 从零开始生成场景 |
| **单张图像** | 将 2D 图像"提升"（lift）为完整 3D 世界 | 单目 3D 重建 |
| **多张图像** | 多视角图像拼合为统一 3D 场景 | 多目重建 |
| **视频** | 多角度短视频合并为世界 | 动态场景静态化 |
| **粗糙 3D 布局** | 盒子/平面 + 导入资产（Chisel 模式） | 结构化创作 |

> **注**：各模态的具体融合机制（如跨模态对齐方式、是否使用统一编码器）**未公开**。

### 核心功能

**Chisel 模式（结构-风格解耦）**
- 将场景创作分解为"结构"（粗糙 3D 几何：盒子、平面）与"风格"（文字 prompt）两个独立维度
- 支持导入已有 3D 资产，结合文字 prompt 进行风格化渲染
- 设计哲学：让创作者掌控空间布局，由模型负责视觉填充

**世界编辑**
- 局部编辑：物体删除与区域修复（inpainting）
- 全局编辑：风格替换（如"将白天改为黄昏"）、结构性修改

**Expansion（扩展）**
- 选定区域后，向外扩展生成新的连续世界内容
- 实现世界的渐进式增长，无需重新生成整体场景

**Composer 模式（世界拼合）**
- 将多个独立生成的世界拼合为大型连续空间
- 支持构建超出单次生成能力的大规模场景

**视频导出**
- 支持像素精确的摄像机路径控制
- 可渲染高质量飞越（flythrough）视频

### 关键设计亮点

1. **3DGS 作为统一中间表示**：不同于直接生成像素或 NeRF 隐式场景，3DGS 提供了可操作的显式几何结构，天然支持局部编辑与资产导入。
2. **结构-风格解耦（Chisel）**：将传统 text-to-3D 的"黑盒生成"拆解为可控的两步流程，降低创作门槛并提升可控性。
3. **Composer 模式的可扩展性**：通过世界拼合而非单次大规模生成，规避了单次生成场景规模受限的问题。
4. **开源渲染器 Spark**：降低下游集成门槛，支持 Web 端直接运行 3DGS 场景。

## RTFM：实时交互渲染

### 架构

RTFM（Real-Time Foundation Model）采用**自回归扩散 Transformer（Autoregressive Diffusion Transformer）**，在帧序列上进行端到端训练，以预测下一帧为核心任务。

核心设计理念源自 **The Bitter Lesson**（苦涩的教训）：

> 不依赖任何显式 3D 表示（无 mesh，无 Gaussian Splat，无深度图），而是让模型从大规模数据中自主学习复杂的视觉效果（反射、阴影、镜头光晕、遮挡等）。

RTFM 的工作方式类似一个 **learned renderer（习得型渲染器）**：

```
历史帧（带 pose） → KV Cache → 隐式世界状态
                                    ↓
当前 pose → Attention 检索 → 生成新视角帧
```

- **输入**：历史观测帧（每帧携带 3D pose 信息）
- **世界表示**：隐式存储于 KV Cache 中（无显式几何结构）
- **输出**：当前 pose 对应的新视角图像帧

硬件要求：单张 **H100 GPU**，以交互帧率运行。

> **注**：具体的 Transformer 规模（层数、参数量）、扩散步数、训练数据来源均**未公开**。

### 空间记忆机制

RTFM 面临的核心挑战：如何在无界世界中漫游，同时避免上下文窗口无限增长。

**Posed Frame（带位姿的帧）**
- 每一帧观测都携带其在 3D 空间中的 **pose**（位置 + 朝向）
- 所有历史帧构成一个空间索引的记忆库（Spatial Memory）

**Context Juggling（上下文调度）**
- 在每一推理步，从空间记忆中检索与当前 pose 空间邻近的帧，动态构建上下文窗口
- 丢弃空间上远离当前视点的历史帧
- 效果：计算开销不随世界探索范围线性增长，实现**无界世界的持久化记忆**

```
空间记忆库（所有历史 posed 帧）
        ↓ 按当前 pose 检索邻近帧
动态上下文窗口（固定大小）
        ↓
Transformer 推理 → 下一帧
```

> **注**：检索策略的具体实现（如 k-NN 检索、空间哈希、记忆压缩方式）**未公开**。

### 关键设计亮点

1. **无显式 3D 表示的 learned renderer**：彻底规避了显式几何重建的误差传播问题，复杂视觉效果（反射、焦散、光晕）通过数据驱动自然涌现。
2. **Context Juggling 实现无界持久化**：以空间邻近性为检索准则的动态上下文管理，是实现大规模世界漫游的关键工程创新。
3. **端到端帧级训练**：以"预测下一帧"为统一目标，无需任何中间监督信号（深度、法线、语义标签），训练流程简洁。
4. **单 GPU 交互帧率**：在保持生成质量的同时，实现了工程上可部署的推理效率。

## 与已有 World Model 范式的关系

当前世界模型领域存在两种主流范式，Marble 与 RTFM 分别对应其中之一：

| 范式 | 代表方法 | 核心表示 | 优势 | 局限 |
|------|---------|----------|------|------|
| **3D Neural** | Marble、NeRF、3DGS | 显式 3D 结构（Gaussian Splats、Mesh） | 几何精确、可编辑、物理兼容 | 重建耗时、静态场景为主 |
| **Generative** | RTFM、GAIA-1、UniSim | 隐式神经表示（KV Cache、latent） | 端到端、泛化强、复杂效果自然涌现 | 几何不可控、难以精确编辑 |

**World Labs 的组合策略代表了一种新趋势**：

> 用 3D Neural 模型（Marble）负责离线的高保真场景重建与结构化编辑，用 Generative 模型（RTFM）负责在线的实时交互渲染——两者通过"3DGS 场景 → 帧序列"的接口衔接，互补而非替代。

这一趋势的深层逻辑在于：纯 Generative 范式缺乏对几何结构的精确控制（难以支持物理仿真和资产复用），而纯 3D Neural 范式在实时交互上存在渲染瓶颈。混合架构将两者优势结合，是当前具身智能场景下世界模型走向实用化的重要路径。

与同期工作的对比：
- **GAIA-1（Wayve）**：纯 Generative，专注自动驾驶场景视频预测，无显式 3D 表示
- **UniSim（Toyota Research）**：基于 NeRF 的场景模拟，侧重物理动力学
- **World Labs**：率先将 3DGS 生成与 Generative 渲染器显式解耦为两个协作模块

## 局限性

1. **架构细节未公开**：Marble 的编码器结构、训练目标、数据集规模，以及 RTFM 的模型参数量、扩散步数、检索实现等核心技术细节均未在博客中披露，无法进行严格的技术复现或对比分析。

2. **静态场景生成为主**：Marble 当前主要针对静态或准静态场景（建筑、室内、自然环境），对动态物体（人、车辆、流体）的生成与编辑能力尚不明确。

3. **RTFM 的单 GPU 限制**：当前 RTFM 在单张 H100 GPU 上运行，消费级硬件的可行性未知；H100 的高成本限制了其在边缘设备和机器人平台上的部署潜力。

4. **物理一致性存疑**：RTFM 作为 learned renderer，其生成的新视角帧在物理上的一致性（如遮挡关系、光照连贯性）依赖训练数据分布，在 out-of-distribution 场景下的表现未有系统评估。

5. **Composer 模式的接缝问题**：多个世界拼合时的边界一致性（几何连续性、光照统一性）是已知难题，博客未提供定量评估。

6. **无正式论文**：目前仅有官方博客，缺乏同行评审，技术声明的可靠性有待学术验证。
